# 编码规范 Skill

基于 Andrej Karpathy 提出的 LLM 编码最佳实践，适用于本项目的前后端开发。

**权衡：** 这些规范偏向谨慎而非速度。简单任务（typo 修复、显而易见的一行代码）不必严格遵循。

---

## 1. 先思考再编码

**不要假设。不要隐藏困惑。暴露权衡。**

开始实现之前：
- **明确说出你的假设**。不确定就问。
- **存在多种解读时，都列出来**——不要 silently 选一种就跑。
- **有更简单的方案就说出来**。该推回就推回。
- **遇到不确定的地方就停下来**。说清楚哪里不懂，然后问。

## 2. 简洁优先

**解决问题的最少代码。不写投机性代码。**

- 不做超出需求的功能
- 只用一次的东西不抽象
- 不加没被要求的"灵活性"或"可配置性"
- 不为不可能的场景写错误处理
- 如果 200 行能缩成 50 行，重写

自问：「一个资深工程师会觉得这段代码过度设计吗？」如果是，简化。

## 3. 精准修改

**只动必须动的。只清理自己制造的垃圾。**

编辑已有代码时：
- 不要"顺手改进"旁边的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你会写得不一样
- 发现不相关的死代码？提一嘴，别删

你的改动产生孤儿代码时：
- 删除**你自己的改动**造成的无用 import/变量/函数
- 不要删除改动之前就存在的死代码（除非被要求）

检验标准：**每一行改动都能追溯到用户的直接需求。**

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

把指令式任务转化为可验证的目标：

| 原始指令 | 转化为 |
|---------|--------|
| "加个校验" | "为无效输入写测试，然后让测试通过" |
| "修这个 bug" | "写一个复现 bug 的测试，然后让测试通过" |
| "重构 X" | "确保重构前后测试都通过" |

多步任务，先列计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强成功标准让 LLM 能独立循环。弱标准（"让它跑起来"）需要不断追问。

## 本项目补充规范

### 技术栈
- **后端：** FastAPI + SQLAlchemy + MySQL，纯 REST API
- **前端：** Vue 3 + TypeScript + Element Plus + Pinia + Vue Router
- **目录：** `backend/` + `frontend/` 分离

### API 规范
- 统一返回格式：`{ "code": 0, "msg": "...", "data": {...} }`
- 分页接口返回：`{ items, total, page, page_size }`
- 认证：JWT Bearer Token

### 前端规范
- 组件用 `<script setup lang="ts">`
- API 请求走 `src/api/` 模块
- 状态管理用 Pinia，不跨模块共享
- 样式用 `<style lang="scss" scoped>`

### 提交规范
- `feat:` 新功能
- `fix:` 修复
- `refactor:` 重构（不改功能）
- `docs:` 文档

---

**这些规范有效的标志：** diff 里没有多余改动、没有因为过度设计而重写、澄清问题在实现之前而非之后、PR 干净精简。
